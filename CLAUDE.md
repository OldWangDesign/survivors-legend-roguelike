# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Godot 4.5 + GDScript Vampire Survivors-style Roguelike (《幸存者传说》). 22 weapon types (15 base + 7 super), 10 characters, 5 maps, Boss every 5 min, supports PC and Android. Working language for code comments, docs, and in-game text is **Chinese**. Read `README.md` for game design overview; this file covers the parts needed to write code productively.

## Running and building

This is a Godot project — there is no `npm`/`make`/test runner.

- **Run from editor**: open `project.godot` in Godot 4.5, press F5. Main scene is `scenes/main_menu.tscn`.
- **Run from CLI** (headless not supported, needs display): `godot project.godot` from the project root.
- **Run a specific scene from CLI**: `godot --path . res://scenes/game.tscn` (skips menu).
- **Android export**: configured in `export_presets.cfg`, output goes to `build/android/survivors_legend.apk`. Trigger via Godot editor's Project → Export, or `godot --headless --export-release "Android" build/android/survivors_legend.apk`.
- **No test suite exists.** Don't invent commands for tests/lint — there's no GDScript linter wired up either.

`.gd.uid` files next to every `.gd` are auto-generated by Godot 4.5's resource system. Never hand-edit; commit them alongside their script.

## Architecture

### Four Autoloads form the runtime backbone

Registered in `project.godot` `[autoload]`. They're globals — refer to them by name from anywhere:

| Autoload | File | Role |
|----------|------|------|
| `GameData` | `scripts/game_data.gd` | Single source of truth for **static config** (`WEAPON_DATA`, `CHARACTER_DATA`, `STAGE_DATA`, level scaling formulas) **and runtime cross-references** (`player_ref`, `enemies_container`, `pickups_container`, `total_kills`, `elapsed_time`). Used as a service locator — most systems read player stats and shared containers through it. |
| `AudioManager` | `scripts/audio_manager.gd` | BGM + SFX. Call `AudioManager.play("name")`. |
| `VfxPool` | `scripts/vfx_pool.gd` | Pooled visual effects (float text, sparks, ring waves, hit flashes). Call `VfxPool.float_text(...)`, etc. — do not instantiate VFX scenes directly. |
| `SpatialGrid` | `scripts/spatial_grid.gd` | **Critical for performance.** O(1) enemy lookups via 128px hash grid. All enemy queries must go through this — see "Performance rules" below. |

### `game.tscn` is built procedurally

`scenes/game.tscn` is mostly empty. `scripts/game_manager.gd` `_ready()` constructs the running game in code: background → containers (Pickups/Enemies) → player → camera → spawner → UI. When adding new in-game systems (chests, world events, etc.), the wiring point is `game_manager.gd`, not the scene tree.

### Weapons follow a strict base-class pattern

All weapons extend `WeaponBase` (`scripts/weapons/weapon_base.gd`) which provides:

- Auto cooldown ticking in `_physics_process` → calls `attack()` when ready.
- `get_damage()` / `get_cooldown()` that automatically apply player multipliers.
- `get_enemies_in_range()` / `get_nearest_enemy()` — already wired to SpatialGrid.
- `play_weapon_sound()`, `spawn_damage_number()`, `start_cooldown()`.

To add a weapon: extend `WeaponBase`, override `attack()`, add an entry to `GameData.WEAPON_DATA` and the `WeaponType` enum, register icon/level scaling.

**Sub-script convention**: many weapons split into a controller + effect pair:

- `fireball.gd` (spawner) + `fireball_explosion.gd` (boom)
- `meteor.gd` + `meteor_strike.gd`
- `holy_water.gd` + `holy_water_zone.gd`
- `lightning.gd` + `lightning_bolt.gd`
- `cross.gd` + `cross_projectile.gd`
- `poison_cloud.gd` + `poison_cloud_zone.gd`
- `freeze_ray.gd` + `freeze_ray_beam.gd`

When adding a similar weapon, follow the same `weapon_name.gd` + `weapon_name_<part>.gd` naming. `projectile.gd` is a generic shared projectile used by several weapons.

### Super weapon synthesis

7 super weapons fuse two base weapons at Lv.8. The fusion rules live in `GameData` (search `WeaponType.INFERNO_STORM` etc. and the level-up offer logic). `level_up_panel.gd` shows golden synthesis options. When changing balance or rules, both data tables and the offer logic must agree.

### Character system

10 characters with passives. Static data in `GameData.CHARACTER_DATA`. Active character is selected in `char_select.tscn` and stored on `GameData`. Passive logic lives mostly in `player.gd` (`get_effective_*` methods compute final stats including passives). Some passives are time-tickers checked in `player.gd`'s `_process`/`_physics_process`.

## Performance rules (mandatory — see `doc/performance_optimization.md`)

This project recently went through a major perf pass to fix late-game frame drops (5 FPS → 25-30 FPS at 8 super weapons + 250+ enemies). Subsequent code must respect these rules or it will regress badly.

1. **Never use `get_nodes_in_group("enemies")` for spatial queries.** Use `SpatialGrid.get_in_range(pos, radius)` / `get_nearby(pos, radius)` / `get_nearest(pos, max_range)`. Group traversal is O(N) and ruins late-game frames.
2. **Persistent-damage weapons must throttle.** For weapons that hit every frame (bible, divine_apocalypse, death_scythe, spin_blade), use a 0.1s `DMG_INTERVAL` accumulator and multiply damage by `DMG_INTERVAL` instead of `delta`. Same DPS, 1/6 the calls.
3. **`queue_redraw()` is conditional.** Don't call unconditionally in `_process`. Enemies redraw only when flashing/elite/wounded; background redraws only when camera moves >16px. Follow the same gating for any new `_draw`-heavy node.
4. **Tree-mutating signals need `call_deferred`.** When responding to `tree_exiting` or other signals fired during scene-tree blocked state, `add_child` will fail silently. Use `call_deferred("_method", args)` to defer to next frame. The fireball→explosion chain is the canonical example.
5. **Enemies must register/update/unregister with SpatialGrid.** New enemy types: `register` in `_ready`, `update_position` after movement (only re-hashes on cell change, so it's cheap to call every frame), `unregister` in `_die` / `queue_free` paths.
6. **Filter `is_instance_valid`** when iterating SpatialGrid results around frames where enemies might die. The grid does this internally for `get_nearby`/`get_in_range`/`get_nearest`, but custom iterations need it.

## Layout reference

```
project.godot                 # Autoloads, display, rendering — edit carefully
scenes/                       # Only 4 scenes — most game content is in scripts
scripts/
  game_data.gd                # Globals + static tables + runtime refs (Autoload)
  game_manager.gd             # game.tscn runtime — wires everything together
  spatial_grid.gd             # Perf-critical, see rules above (Autoload)
  audio_manager.gd            # (Autoload)
  vfx_pool.gd                 # (Autoload)
  player.gd                   # Stats, passives, input
  enemy.gd / boss_enemy.gd    # Enemy AI; both register with SpatialGrid
  enemy_spawner.gd            # Wave spawning during a stage
  enemy_formation.gd          # Group/formation patterns for enemies
  stage_spawner.gd            # Per-stage wave config (referenced by enemy_spawner)
  weapons/                    # 22 weapons, all extend weapon_base.gd
  ui/                         # All UI scripts; HUD, menus, panels
  vfx/                        # VFX node implementations used by VfxPool
  world/                      # World events: danger_zone, healing_point
doc/                          # Design docs and PRDs (Chinese)
assets/                       # Sprites/bg/audio/fonts/icons
build/                        # Android export output
```

## Conventions

- **Comments and string literals**: Chinese is the project language. Don't translate existing Chinese comments to English.
- **No type-strict mode** is enforced, but most files use type hints (`var x: int`, `-> Vector2`). Match the surrounding style.
- **Scene-tree-light, code-heavy.** Most game objects (player, enemies, UI) are constructed in code via `set_script(preload(...))`, not as `.tscn` instances. Don't try to find a `Player.tscn` — there isn't one.
